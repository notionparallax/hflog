# Dive Human Factors Log — System Brief v2

**Status:** Phase 1 complete (deployed PWA prototype). Phase 2: design / pre-build.
**Repo:** https://github.com/notionparallax/hflog
**Live:** https://notionparallax.github.io/hflog/

---

## Purpose

A **confidential** (not anonymous) reporting system that captures structured human-factors metadata for **every dive**, so that near-miss and incident **rates** can be established against a real denominator.

Two design facts drive everything:

1. **Confidential, not anonymous.** The diver is never named to the public or in any analytical export. Identity is held securely by the operating body so that, for a serious or instructive report, the diver can be asked to make contact. De-identification is a hard boundary, not a display filter.
2. **Every dive logged = real denominator.** The system is only analytically useful if uneventful dives are logged too. Nil-event logging must be fast enough that divers actually do it. The count of logged dives ≈ the true number of dives, which allows genuine near-miss and incident *rates* rather than raw counts.

## Phase 1 — Data Collection PWA ✓ Complete

### What was built

A Svelte 5 Progressive Web App (PWA) that:

- Presents a structured human-factors form optimised for mobile use poolside
- Produces a versioned, machine-readable text block (`[DIVE-HF v1] … [/DIVE-HF]`)
- The block is copied and pasted by the diver into the **Notes** field of their dive in their log software (e.g. Subsurface). The log software is the carrier; no modification to it is required.
- Works fully offline after first load (service worker caches all assets)
- Is installable as a home-screen app on iOS and Android

### Architecture & stack

| Concern | Technology |
|---|---|
| Framework | Svelte 5 (runes) + Vite 8 |
| Language | JavaScript (no TypeScript) |
| Deployment | GitHub Pages via GitHub Actions CI |
| Offline | Workbox service worker (auto-generated by vite-plugin-pwa) |
| Tests | Vitest — serializer round-trip and field coverage |
| Backend | **None** — all intelligence in the SPA; block is the transport |
| Storage | **None** — ephemeral; state resets on page load |

### Data model (as built)

All controlled vocabularies live in `src/lib/vocab.js` — the single place to edit when taxonomies change.

#### Pre-dive context (every dive)

| Field | Type | Scale / values |
|---|---|---|
| `report_id` | UUID | Auto-generated per session |
| `imsafee.illness` | int | 1–5 (see anchors) |
| `imsafee.medication` | int | 1–5 |
| `imsafee.stress` | int | 1–5 |
| `imsafee.alcohol` | int | 1–5 |
| `imsafee.fatigue` | int | 1–5 |
| `imsafee.eating` | int | 1–5 |
| `imsafee.emotion` | int | 1–5 |
| `site_familiarity` | enum | `first_time`, `return`, `regular` |
| `buddy_familiarity` | enum | `solo`, `stranger`, `occasional`, `regular` |
| `brief_quality` | int | 1–5 (no brief → full written plan) |
| `time_pressure` | int | 1–5 |
| `task_complexity` | int | 1–5 (routine → beyond limits) |
| `plan_adherence` | int | 1–5 (exact → substantially departed) |
| `new_kit` | bool + str | false or "description of new/changed equipment" |

**IMSAFEE** is the aviation pre-flight readiness check (Illness, Medication, Stress, Alcohol, Fatigue, Eating, Emotion) adapted for diving. The standard IMSAFE binary check is extended to a 1–5 scale for research granularity. This introduces inter-rater variance; anchor calibration exercises are recommended before cross-diver aggregation is treated as quantitative.

#### Events (zero or more per dive)

Each dive may record multiple events. `had_event` is derived (`events.length > 0`), not entered.

| Field | Type | Notes |
|---|---|---|
| `event_N_type` | enum | See vocabulary below; DIRS-aligned |
| `event_N_description` | str | Free text |
| `event_N_severity` | int 1–5 | **Axis A** — how bad could the outcome have been? |
| `event_N_proximity` | int 1–5 | **Axis B** — how close did it actually come? |
| `event_N_outcome` | enum | See vocabulary below |

**Severity and proximity are two orthogonal axes — never collapse them into one number.** Both are needed to distinguish a catastrophic near-miss (5,5) from a trivial incident (1,1) and an averted catastrophe (5,1).

#### Classification (derived)

| Value | Condition |
|---|---|
| `nil` | No events logged |
| `near_miss` | ≥1 event, all outcomes `no_harm_near_miss` |
| `incident` | ≥1 event with outcome other than `no_harm_near_miss` |

#### Human factors & learning

| Field | Type | Notes |
|---|---|---|
| `factors[]` | enum[] | Multi-select; see vocabulary below |
| `learning` | str | Free text — personal decision-making + systemic factors |
| `what_went_well` | str | Positive capture: protective behaviours and planning |

#### Reflection prompts (conditional)

Contextual questions triggered by elevated scores (IMSAFEE ≥ 4, severity ≥ 3, proximity ≥ 4). Up to 4 Q&A pairs per block. Stored as `reflection_N_q` / `reflection_N_a`.

### Controlled vocabularies

#### Event types (DIRS-aligned)

`rapid_ascent`, `out_of_gas`, `buoyancy_loss`, `valve_not_fully_open`, `entanglement`, `lost_guideline`, `silt_out`, `light_failure`, `gas_switch_error`, `dcs_symptom`, `equipment_failure`, `buddy_separation`, `navigation_error`, `other`

Vocabulary is aligned to DAN DIRS (Diving Incident Reporting System) to support future data pooling with national and international datasets.

#### Outcomes

`no_harm_near_miss`, `minor_injury`, `treatment_required`, `evacuation`, `fatality`

#### Factors (15 — Dirty Dozen + systemic tier)

Individual: `communication`, `complacency`, `knowledge`, `distraction`, `teamwork`, `fatigue`, `resources`, `pressure`, `assertiveness`, `stress`, `awareness`, `norms`, `cold`

Systemic: `training_gap`, `equip_design`, `org_pressure`

The systemic tier points upstream past the individual. Selecting a systemic factor triggers reflection prompts about the organisational or design conditions that enabled the event.

### DIVE-HF v1 Block Format Specification

The block lives inside the dive log Notes field, coexisting with human free text. Everything between the sentinels is machine-readable; everything outside is preserved verbatim.

#### Grammar

```
block      := SENTINEL_OPEN LF field* SENTINEL_CLOSE
SENTINEL_OPEN  := "[DIVE-HF v" DIGIT+ "]"
SENTINEL_CLOSE := "[/DIVE-HF]"
field      := key ": " value LF
key        := [a-z_0-9]+
value      := bool | number | quoted_str | inline_obj | inline_array
bool       := "true" | "false"
number     := "-"? DIGIT+ ("." DIGIT+)?
quoted_str := '"' (escaped_char | [^"\n])* '"'
inline_obj := "{" (key ": " value ", ")* key ": " value "}"
inline_array := "[" (bare_value ", ")* bare_value "]"
```

Quoted strings escape `"` as `\"` and `\` as `\\`. Newlines in string values are normalised to spaces.

#### Complete field reference (v1)

```
[DIVE-HF v1]
report_id: <uuid4>
had_event: <true|false>
imsafee: {illness: N, medication: N, stress: N, alcohol: N, fatigue: N, eating: N, emotion: N}
site_familiarity: <first_time|return|regular|"">
buddy_familiarity: <solo|stranger|occasional|regular|"">
brief_quality: N
time_pressure: N
task_complexity: N
plan_adherence: N
new_kit: <false|"description">
                                    ← event fields only present when had_event: true
event_count: N
event_1_type: <event_type>
event_1_description: "..."          ← optional
event_1_severity: N
event_1_proximity: N
event_1_outcome: <outcome>
event_2_type: ...                   ← repeat for each event
classification: <nil|near_miss|incident>
factors: [factor, ...]              ← omitted if empty
learning: "..."                     ← omitted if empty
what_went_well: "..."               ← omitted if empty
reflection_1_q: "..."               ← omitted if no reflections triggered
reflection_1_a: "..."               ← omitted if not answered
[/DIVE-HF]
```

All numeric scales are 1–5. A value of 1 always represents the least-concern / most-nominal state.

#### Parsing rules

- Locate block by `[DIVE-HF vN]` … `[/DIVE-HF]` sentinels.
- Preserve all text outside sentinels exactly (both before and after).
- Warn but do not throw on unknown schema version N.
- `event_N_*` fields with the same N form a single event object (1-indexed).
- `had_event` is derived in the parser from the presence of `event_count`; the serialised value is informational.
- Old `[CDAA-HF v1]` blocks (pre-v2 prototype) should be accepted with a deprecation warning.

## Phase 2 — Aggregation, Identity & Analysis Backend

### Motivation

Phase 1 produces blocks inside individual dive logs. Phase 2 collects them, stores them securely, and makes the denominator meaningful — turning individual records into a population dataset.

The analytical payoff is only reachable when:
1. Enough divers are logging regularly that the denominator is real
2. A backend can aggregate across divers without exposing individual identity
3. An analyst can see rates, patterns, and risk signals

### Proposed architecture

| Concern | Technology |
|---|---|
| Backend | Python 3.12 + FastAPI |
| Storage | SQLite (single-file, portable) via SQLAlchemy |
| Identity boundary | Two tables: `reports` (analytical) + `identities` (contact only) |
| Subsurface ingestion | Python CLI — parses SSRF/XML, extracts DIVE-HF blocks |
| Direct submission | Optional API endpoint from SPA (requires simple auth token) |
| Export | CSV + JSON, selecting only from `reports` table |
| Dashboard | Server-rendered (Jinja2) or extend the existing Svelte SPA |

### Two ingestion paths

**Path A — Subsurface import (existing workflow):**
Diver exports their Subsurface log as XML → CLI tool parses all `<notes>` fields, extracts DIVE-HF blocks, inserts into database. All dives in the log count as denominator regardless of whether a block is present.

**Path B — Direct SPA submission (optional):**
Add a "Submit" button to the SPA that POSTs the serialised block to an API endpoint. Requires a per-diver auth token (no password flow needed — a static bearer token per reporter is sufficient for prototype). Enables real-time aggregation without Subsurface export step.

Both paths write to the same `reports` table. A diver may use both; deduplication is on `report_id`.

### Confidentiality boundary (unchanged from original brief)

Model identity as a **hard boundary**, not a filter:

- `reports` holds all analytical fields keyed by `reporter_id` (opaque handle). Nothing in this table identifies a person.
- `identities` holds `reporter_id → contact_detail`, `contactable`, `follow_up_requested`. Nothing else joins to this table.
- The export function selects **only** from `reports`. A test asserts the export contains zero contact fields.
- An analyst view may set `follow_up_requested` on a report; resolving that to an actual contact detail is a separate, access-gated CLI command.

### Aggregation dashboard (minimum viable)

From the ingested data, compute and display:

| Metric | Notes |
|---|---|
| **Denominator** | Total logged dives |
| **Numerator** | Dives with `had_event: true`; incidents vs near-misses separately |
| **Rate** | Near-misses per 100 dives; incidents per 100 dives |
| **Severity × proximity matrix** | 5×5 heatmap count |
| **Factor frequency** | Bar chart of `factors[]` |
| **Pre-dive signal** | Cross-tab of `had_event` against IMSAFEE fields and the new sliders (brief_quality, task_complexity, plan_adherence) |
| **What went well** | Free-text corpus for qualitative review |

Display a visible caveat that early figures are unstable and reporting-culture-sensitive.

### Phase 2 non-goals

- Production multi-user auth (KMS, OAuth, MFA)
- Cloud hosting or multi-org tenancy
- Statistical rigour (confidence intervals, under-reporting correction models) — leave hooks, do not implement
- Any modification to Subsurface itself
- Git-backed Subsurface storage format

### Future work (post Phase 2)

- Export mapper to DAN DIRS / DISMS schema for data pooling
- Statistical inference layer (Bayesian rate estimation, under-reporting model)
- Analyst review workflow with access-gated contact resolution
- Multi-organisation tenancy
- Structured debrief workflow (DEBRIEF framework integration)

## Notes for builders

- All controlled vocabularies live in one place (`src/lib/vocab.js` in Phase 1; mirror to `vocab.py` in Phase 2). Expect them to change — do not hardcode elsewhere.
- The two-axis severity × proximity model is a hard design constraint. Never derive a single risk score from both axes.
- Anchor text must be displayed in the UI alongside every scale. Inter-rater consistency depends on it.
- The `what_went_well` field is as analytically important as the event fields — it surfaces protective factors that would otherwise be invisible in a near-miss-only dataset.
- The systemic factor tier (`training_gap`, `equip_design`, `org_pressure`) was added to address the individual-blame tendency of the Dirty Dozen taxonomy. Trends in these factors point at organisational rather than individual causes.
